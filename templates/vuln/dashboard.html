Vulnerability Lab

An intentionally vulnerable application built as a WAF testing target. Six isolated modules, each exposing a distinct OWASP Top 10 attack surface.

← Back to Dashboard
01 Critical
SQL Injection
Login form built with raw SQL string formatting — no parameterized queries. Accepts payloads like ' OR 1=1 -- to bypass auth and extract the full users table.
/vuln/sqli ↗
02 High
Cross-Site Scripting
Search input reflected directly into the page using Jinja2's | safe filter. Accepts <script> tags, event handlers, and SVG injection payloads.
/vuln/xss ↗
03 High
Unrestricted File Upload
File upload endpoint with no MIME type, extension, or filename validation. Accepts .php, .exe, .sh and saves directly to the server filesystem.
/vuln/upload ↗
04 Critical
Command Injection
Host input passed directly into a shell command via subprocess. Shell metacharacters like ; id, && cat /etc/passwd, and $() execute freely.
/vuln/cmdi ↗
05 Medium
Broken Access Control
Admin panel with zero authentication or authorization checks. Any anonymous visitor gets full access to all user records including plaintext credentials.
/vuln/admin ↗
06 High
Server-Side Request Forgery
Any user-supplied URL is fetched server-side via requests.get(). Probe internal services, cloud metadata endpoints, and private network ranges.
/vuln/ssrf ↗

WAF Testing Flow — Place a WAF in front of this app (Nginx + ModSecurity, AWS WAF, Cloudflare, HAProxy + CRS). Send attack payloads through it and verify detection, logging, and blocking behavior. This application performs no input validation — that responsibility belongs entirely to the WAF.
